# 混合推荐算法与 AI 店铺数据分析设计说明

## 1. 文档目标

本文档用于补充项目中两项新增智能化功能的实现细节，重点说明：

1. 前台混合推荐算法是如何采集行为数据、生成候选集、计算分数并做加权融合的。
2. 后台 AI 店铺数据分析是如何从业务数据中取数、如何组织 Prompt、如何要求大模型结构化输出，以及如何兼容不同厂商的大模型接口。

这份说明既可用于开发交接，也可直接作为答辩时“实现方式”部分的技术支撑材料。

## 2. 功能目标

为避免系统停留在传统 CRUD 层面，本项目新增两个智能化模块：

1. 前台混合推荐算法：根据用户浏览、加购、购买记录生成个性化图书推荐。
2. 后台 AI 店铺数据分析：根据订单、营收、库存、热销图书等经营数据生成摘要、预测和运营建议。

## 3. 前台混合推荐算法

### 3.1 设计目标

推荐模块需要同时解决以下问题：

1. 已登录用户要看到带有个性化特征的图书推荐，而不是固定热门榜。
2. 新用户、匿名用户、行为很少的用户也必须有推荐结果，不能出现大片空白。
3. 推荐结果需要可解释，前端能够展示“为什么推荐给我”。
4. 实现方式要足够稳定，优先使用数据库可计算、可落地、可演示的算法。

因此项目没有只使用单一算法，而是采用“多路召回 + 分数归一化 + 加权融合”的混合推荐方案。

### 3.2 行为数据采集

推荐算法依赖用户行为数据。项目新增了 `user_behavior` 表，字段包括：

- `user_id`：用户 ID。
- `book_id`：图书 ID。
- `behavior_type`：行为类型，当前支持 `view`、`cart`、`purchase`。
- `score`：行为基础权重分。
- `create_time`：行为发生时间。

当前行为权重设计如下：

- `view`：浏览图书详情，权重 `1`。
- `cart`：加入购物车，权重 `5`。
- `purchase`：购买行为，权重 `10`。

这样设计的原因是：浏览代表弱兴趣，加购代表中强兴趣，购买代表最强兴趣。系统通过不同权重让“真正成交过的图书”对用户画像贡献更大。

### 3.3 行为数据写入时机

用户行为并不是由定时任务离线导入，而是直接嵌入业务流程中实时写入：

1. 用户进入图书详情页时，前端调用 `POST /api/front/recommendations/behaviors` 上报 `view`。
2. 用户加入购物车时，后端购物车服务在成功写入购物车后记录 `cart`。
3. 用户支付成功后，后端订单服务根据已支付订单的商品列表批量记录 `purchase`。

这样做的优点有三点：

1. 不需要额外埋点平台，系统内即可闭环。
2. 推荐数据实时可用，演示时马上能看到推荐变化。
3. 数据来源与业务操作强绑定，便于解释推荐依据。

### 3.4 兼容旧库的表结构迁移

为了保证系统能够在旧数据库上平滑升级，项目没有要求手工改表，而是在 `SchemaMigrationRunner` 中做了启动自检：

1. 先读取当前数据库名。
2. 查询 `information_schema.TABLES` 检查 `user_behavior` 是否存在。
3. 若不存在，则自动执行 `CREATE TABLE user_behavior ...`。

这样做的意义是：

1. 降低部署门槛。
2. 便于答辩现场直接启动老库演示。
3. 避免因为缺少推荐行为表而导致接口报错。

### 3.5 推荐服务总体流程

推荐主流程位于 `RecommendationServiceImpl`，整体过程如下：

1. 先根据请求参数确定返回数量 `limit`，并限制最大值，避免一次取过多推荐。
2. 根据 `limit` 放大候选集查询数量，当前使用 `candidateLimit = limit × 4`。
3. 分别从四个推荐策略中查询候选图书。
4. 对每一路候选分数做归一化处理。
5. 按预设权重融合到同一候选池。
6. 对同一本书进行去重合并。
7. 按最终融合分从高到低排序。
8. 返回图书基础信息、推荐分数、命中策略和推荐理由。

这里的核心思想不是“谁分高就直接返回”，而是“让多种推荐信号共同参与，单一路径不过度主导结果”。

### 3.6 四路推荐策略

当前系统采用四路候选融合，权重如下：

- 协同过滤：`40%`
- 内容相似：`25%`
- AI 智能重排：`20%`
- 热门趋势：`15%`

总权重为 `100%`。这种配比的设计思路是：

1. 协同过滤最能体现“和你相似的人还买了什么”，因此占比最高。
2. 内容相似能补充图书属性层面的相关性，适合作为第二信号。
3. AI 智能重排负责对兴趣强度、时间衰减和价格区间做更细粒度的修正。
4. 热门趋势主要承担冷启动、匿名访问和兜底任务，因此权重最低，但不能缺失。

### 3.7 协同过滤实现方式

协同过滤不是直接使用复杂矩阵分解，而是使用更适合课程项目和数据库实现的“基于共同购买用户的邻域协同过滤”。

实现步骤如下：

1. 先收集当前用户相关图书集合：
   - `user_behavior` 中出现过的图书；
   - 购物车中的图书；
   - 已下单且状态为待发货、已发货、已完成的订单图书；
   - 如果是图书详情页，还会把当前 `sceneBookId` 合并进集合。
2. 在订单表中寻找“购买过这些图书的其他用户”。
3. 再查询这些相似用户购买过、但当前用户还没有接触过的其他图书。
4. 计算候选书的基础分：
   - `SUM(oi2.quantity) * 5`
   - `COUNT(DISTINCT o2.user_id) * 3`
5. 以基础分高低排序，作为协同过滤候选集。

可把它理解为：

- 买的人越多，说明这本书在相似群体中越有代表性。
- 被相似用户购买的次数越多，说明相关性越强。

虽然不是工业级矩阵算法，但它具备明显的协同过滤特征，且适合当前数据规模。

### 3.8 内容相似推荐实现方式

内容相似推荐基于图书本身属性做匹配，主要使用：

1. 分类 `category_id`
2. 作者 `author`
3. 出版社 `publisher`

实现思路如下：

1. 从用户历史图书中提取偏好画像，得到其常见分类、作者、出版社集合。
2. 在图书表中查找与这些属性相似的在售图书。
3. 用属性命中情况计算内容分。

当前 SQL 中的分数规则为：

- 分类相同：`+8`
- 作者相同：`+5`
- 出版社相同：`+3`
- 若图书是 30 天内新上架：额外 `+2`

最终对同一本候选书的多次属性命中累计求和，作为内容相似分。

这种实现的优点是：

1. 算法简单稳定。
2. 推荐理由容易解释。
3. 对用户行为较少但已有一点偏好的情况也有效。

### 3.9 AI 智能重排实现方式

这里的“AI 智能重排”不是调用外部大模型，而是使用一组更贴近用户兴趣强度的规则特征，对候选书进行再打分。

它的目标不是独立生成推荐，而是在“用户已有兴趣画像”的基础上做更细的排序优化。

重排特征包括：

1. 兴趣强度：不同行为的权重不同，购买高于加购，加购高于浏览。
2. 时间衰减：最近 7 天行为权重更高，30 天内次之，更早行为衰减。
3. 属性相似度：分类、作者、出版社是否与用户偏好匹配。
4. 价格带相似：候选图书价格是否落在用户偏好价格附近。
5. 库存健康度：库存较充足的图书适度加分。
6. 短期热度：近 14 天内的用户行为热度也参与重排。

当前实现中的关键规则如下：

#### 1. 用户兴趣强度

从三类行为合并兴趣信号：

- 浏览行为：`ub.score × 时间衰减系数`
- 购物车行为：`quantity × 5`
- 历史订单行为：`quantity × 10`
- 当前详情页图书 `sceneBookId`：额外补一个固定信号 `8`

#### 2. 时间衰减

`user_behavior` 的时间衰减系数如下：

- 最近 7 天：`1.5`
- 最近 30 天：`1.0`
- 更早：`0.5`

这意味着系统更重视近期兴趣，而不是无限累加过去很久的行为。

#### 3. 候选书属性分

候选书与偏好画像匹配时加分：

- 分类相同：`+12`
- 作者相同：`+6`
- 出版社相同：`+4`
- 价格位于偏好价格的 `0.75 ~ 1.35` 区间：`+3`

#### 4. 热度和库存修正

- 近 14 天行为热度分乘以 `0.25` 叠加。
- 库存 `>= 10`：`+3`
- 库存 `> 0 且 < 10`：`+1`
- 14 天内新书：`+2`

因此这一策略本质上是一个“多特征线性打分模型”，虽然未用机器学习训练，但已经具备“重排模型”的结构。

### 3.10 热门趋势推荐实现方式

热门趋势承担三类职责：

1. 未登录用户的默认推荐。
2. 行为较少用户的冷启动补充。
3. 其他推荐策略候选不足时的兜底结果。

它的实现基于全站热度统计，而不是用户个体画像。分数组成如下：

1. 销量分：
   - `SUM(oi.quantity) * 5`
   - `COUNT(DISTINCT oi.order_id) * 2`
2. 行为热度分：
   - 近 30 天 `user_behavior` 的 `SUM(score)` 再乘以 `0.3`
3. 库存修正：
   - 有库存则额外 `+2`

这样可以避免推荐给用户“虽然热但没货”的图书。

### 3.11 候选集合并与分数归一化

不同策略的原始分数数量级并不一致：

1. 协同过滤可能因为销量累计而很大。
2. 内容相似通常是十几分到几十分。
3. 重排策略会受到兴趣强度和热度累加影响。
4. 热门趋势又是全站热度口径。

如果直接相加，会导致某一路因为量纲更大而碾压其他策略，因此项目在服务层做了归一化处理。

当前归一化方式为：

1. 对每一路候选集单独找出最大分 `maxScore`。
2. 对候选图书做：

   `normalizedScore = (candidateScore / maxScore) × strategyWeight × 100`

3. 将归一化后的结果累加到统一候选池。

这样做的效果是：

1. 每一路策略的最终贡献上限由其权重控制。
2. 不同策略之间的量纲被拉齐。
3. 同一本书可以同时命中多路策略，并获得叠加分。

### 3.12 去重、排序与可解释输出

推荐结果在合并过程中，使用 `bookId` 作为唯一键去重。若同一本书命中多种策略，则：

1. `score` 累加。
2. `strategy` 记录所有命中的策略名称，例如：
   - `协同过滤 + AI智能推荐 + 热门趋势`
3. `reason` 拼接所有命中的推荐理由。

最终返回的 `RecommendationBookVO` 中，除图书基础字段外，还包含：

- `score`：最终融合后的推荐分。
- `strategy`：命中的推荐策略。
- `reason`：可解释推荐理由。

这样前端不仅能展示推荐结果，还能告诉用户“为什么是这本书”。

### 3.13 登录态、匿名态与场景推荐

推荐接口支持两种重要能力：

#### 1. 匿名访问

`/api/front/recommendations` 被拦截器排除在登录校验之外，因此匿名用户也能访问。此时：

1. 不执行个性化协同过滤、内容相似、重排。
2. 主要返回热门趋势推荐。

#### 2. 场景推荐

图书详情页会传入 `sceneBookId`，用于构造“看了又看”场景。该参数会：

1. 进入用户图书集合。
2. 参与内容相似与重排画像。
3. 在查询中排除当前图书自身，避免推荐自己给自己。

因此首页推荐与详情页推荐虽然共用同一接口，但语义不同：

1. 首页偏“猜你喜欢”。
2. 详情页偏“与当前图书相关的继续浏览推荐”。

### 3.14 方案优点与边界

#### 优点

1. 算法完全落地在现有 Spring Boot + MyBatis 架构中，部署简单。
2. 不依赖额外推荐中台或离线训练任务。
3. 兼顾个性化、相似性、热度和兜底。
4. 推荐结果具备可解释性。

#### 当前边界

1. “AI 智能重排”本质上是规则重排，不是在线大模型推荐。
2. 当前行为类型较少，尚未加入搜索词、停留时长、评分、收藏等信号。
3. `purchase` 行为已调整为支付成功后记录，能避免未支付订单污染推荐结果。

## 4. 后台 AI 店铺数据分析

### 4.1 设计目标

后台 AI 分析模块的目标不是简单把图表搬到页面上，而是进一步输出：

1. 面向经营者的摘要结论。
2. 关键指标解释。
3. 下周经营预测。
4. 可执行的运营建议。

同时，系统必须满足三个工程要求：

1. 没有 API Key 时也要可用。
2. 模型超时、异常时页面不能崩。
3. 输出必须结构稳定，前端不能依赖模型“自由发挥”。

因此系统采用“大模型优先 + 规则兜底 + 结构校验”的方案。

### 4.2 数据来源与取数方式

AI 分析本身不直接从每张业务表零散查数，而是先复用后台看板服务 `DashboardService` 聚合出 `AdminDashboardVO`，再将聚合结果喂给 AI 模块。

这样做的好处是：

1. AI 模块只关心经营视图，不直接耦合底层明细表结构。
2. 看板和 AI 分析使用同一份业务口径，避免数字不一致。
3. 后续若扩展数据源，只需要扩展看板聚合层。

当前 AI 分析使用的数据主要包括：

1. 店铺总览：
   - 用户数
   - 管理员数
   - 图书总数
   - 上架图书数
   - 总订单数
   - 待发货订单数
   - 低库存图书数
   - 启用轮播图数
   - 今日订单数
   - 总营收
   - 今日营收
2. 近 7 天订单趋势：
   - 每日订单量
   - 每日营收
3. 订单状态分布：
   - 待支付、待发货、已发货、已完成、已取消等数量
4. 分类统计：
   - 各分类图书数量
5. 热销图书列表：
   - 图书名、作者、销量、销售额
6. 低库存图书列表
7. 最近订单列表

### 4.3 数据聚合与规范化

后台看板层已经先做了第一轮业务规范化：

1. 统一以 `AdminDashboardVO` 作为 AI 输入对象。
2. 趋势数据固定为最近 7 天。
3. SQL 查询只返回有数据的日期时，服务层会自动补齐缺失日期为 `0`，保证折线图和 AI 分析都能看到连续时间序列。
4. 热销图书封面地址在服务层统一补全为可访问 URL。

这种规范化处理对 AI 分析非常关键，因为模型更擅长处理“口径稳定、字段一致”的结构数据，而不适合直接面对零散 SQL 明细。

### 4.4 AI 分析主流程

后台 AI 分析服务 `AdminAiAnalysisServiceImpl` 的主流程如下：

1. 调用 `dashboardService.getAdminDashboard()` 读取当前经营聚合数据。
2. 先构建一份本地规则分析结果 `fallback`。
3. 若配置允许，则尝试调用外部大模型。
4. 若模型返回成功且 JSON 结构合法，则使用模型结果。
5. 若模型未开启、超时、异常、状态码异常或返回结构不完整，则直接返回规则兜底结果。

这里的关键不是“必须调用大模型”，而是“无论大模型是否可用，都能稳定返回结构化分析结果”。

### 4.5 大模型调用触发条件

模型调用由以下配置共同控制：

- `ai.model.enabled=true`
- `ai.model.api-key` 非空
- `ai.model.base-url` 配置正确
- `ai.model.model` 指定了可用模型

只要 `enabled=false` 或 `api-key` 为空，客户端就直接返回 `disabled`，服务层自然会走规则兜底。

也就是说，项目默认部署状态更偏向“本地规则分析”，开启 API Key 后才升级为“真 AI + 规则兜底”的模式。

### 4.6 OpenAI 兼容协议的实现方式

项目中的 `OpenAiCompatibleClient` 没有绑定某一家模型厂商，而是直接实现了 OpenAI Chat Completions 兼容协议的 HTTP 调用逻辑。

请求体主要字段包括：

- `model`
- `temperature`
- `max_tokens`
- `response_format: { "type": "json_object" }`
- `messages`
  - `system`
  - `user`

请求方式是标准的：

1. 使用 Java `HttpClient` 发起 `POST` 请求。
2. 请求头设置：
   - `Content-Type: application/json`
   - `Authorization: Bearer {apiKey}`
3. 请求地址规则：
   - 若 `baseUrl` 已经是完整的 `/chat/completions` 或 `/responses`，直接使用；
   - 否则自动在 `baseUrl` 后拼接 `/chat/completions`。

因此它可以兼容：

1. OpenAI 官方接口
2. 阿里云百炼/通义千问兼容接口
3. 自建 OpenAI 兼容网关
4. 其他遵循同一协议的模型服务

这种兼容方式的优点是：

1. 不需要为每家模型厂商单独写 SDK。
2. 配置改动即可切换模型。
3. 后续接入成本低。

### 4.7 Prompt 设计与输出约束

为了避免大模型自由发挥导致前端难以解析，系统把 Prompt 设计成“强约束结构化输出”。

#### 1. System Prompt 约束

系统角色提示要求模型：

1. 你是图书商城后台经营分析专家。
2. 必须只输出合法 JSON。
3. 不要 Markdown。
4. 不要代码块。
5. 只能基于提供的数据分析，不能编造不存在的数据。

#### 2. User Prompt 约束

用户提示进一步规定：

1. 输出必须是 JSON 对象。
2. 字段固定为：
   - `summary`
   - `metrics`
   - `forecast`
   - `insights`
3. `metrics` 必须是 4 项。
4. `insights` 必须是 3 到 5 项。
5. `level` 只能是：
   - `success`
   - `warning`
   - `info`
6. `summary` 长度限制在 60 字以内。
7. 每条 `insight` 和 `action` 都有字数限制。
8. 金额保留两位小数。
9. 预测必须保守。

最后，系统把 `AdminDashboardVO` 序列化成 JSON 拼进 Prompt 中，作为模型唯一数据来源。

### 4.8 结构化输出与服务端校验

项目没有盲目信任模型输出，而是做了两层控制：

#### 第一层：请求阶段约束

通过 `response_format = json_object` 要求模型直接生成 JSON。

#### 第二层：服务端解析校验

收到模型文本后，后端会：

1. 解析为 `JsonNode`。
2. 提取 `summary`。
3. 提取 `metrics` 数组。
4. 提取 `forecast` 对象。
5. 提取 `insights` 数组。
6. 检查关键字段是否缺失或为空。
7. 限制指标数最多 4 项，建议数最多 5 项。
8. `level` 非法时自动归一化为 `info`。
9. 金额和数字字段解析失败则视为无效。

只要任何关键结构不满足要求，系统就直接放弃模型结果，返回本地规则分析结果。

这一步非常重要，因为它保证了：

1. 前端模板字段始终稳定。
2. 模型偶发跑偏不会污染页面。
3. 管理端页面可以放心使用固定渲染逻辑。

### 4.9 规则兜底分析的实现方式

规则兜底不是简单写一段固定文案，而是基于当前经营数据动态生成的。

主要步骤如下：

1. 从近 7 天趋势中累计得到：
   - `weekOrders`
   - `weekRevenue`
2. 计算客单价：
   - `avgOrderValue = weekRevenue / weekOrders`
3. 计算近 7 天增长率：
   - 将 7 天拆成前半周与后半周
   - `growthRate = (secondHalf - firstHalf) / firstHalf`
4. 基于增长率预测下周订单和营收。
5. 根据库存、待发货、爆款图书、客单价等条件生成建议。
6. 组装摘要、指标、预测、建议和重点图书。

这说明即使没有外部模型，系统也不是“假装 AI”，而是至少具备一套规则分析引擎。

### 4.10 指标、预测与建议的生成逻辑

#### 1. 指标生成

当前规则版固定输出 4 个指标：

1. 近 7 天订单
2. 近 7 天营收
3. 客单价
4. 库存预警

每个指标都包含：

- `label`
- `value`
- `trend`
- `hint`

也就是说，前端拿到的不是裸数字，而是“指标 + 趋势标签 + 解释说明”的组合。

#### 2. 预测生成

下周预测使用的是“基于增长率的线性外推”，具体公式为：

1. `factor = clamp(1 + growthRate, 0.75, 1.35)`
2. `nextOrders = round(weekOrders × factor)`
3. `nextRevenue = weekRevenue × factor`

其中 `clamp` 的作用是防止预测过于夸张：

- 最低不低于当前周的 `75%`
- 最高不高于当前周的 `135%`

同时系统还会根据订单量给出置信度：

- 订单数 >= 10：高
- 订单数 >= 3：中
- 否则：低

#### 3. 运营建议生成

规则建议主要覆盖以下几类经营场景：

1. 趋势场景：
   - 订单明显下滑
   - 订单明显上升
   - 订单总体平稳
2. 库存场景：
   - 存在低库存图书
3. 履约场景：
   - 存在待发货订单
4. 爆款场景：
   - 存在热销第一名图书
5. 客单价场景：
   - 客单价偏低

系统会根据命中的条件拼装多条建议，并为每条建议给出：

- `type`
- `level`
- `title`
- `content`
- `action`

### 4.11 超时、异常与失败处理

模型调用在工程上做了较完整的异常处理：

1. 未启用或未配置 API Key：直接 `disabled`
2. HTTP 非 2xx：`failed`
3. 请求超时：`timeout`
4. 中断异常：`timeout`
5. JSON 解析失败：`failed`
6. 模型返回空内容：`failed`

服务层只要发现不是完整成功，就会回退到本地规则结果。

同时，前端还会根据 `source` 和 `sourceLabel` 显示：

1. `model`：AI 模型生成
2. `rule`：规则兜底
3. `timeout`：AI 超时兜底

这使得系统在演示时也能清楚说明“这次结果来自哪里”。

### 4.12 前端渲染与缓存策略

后台 `AiAnalysis.vue` 页面并不会一进页面就自动调用模型，而是采用“手动触发分析”的方式：

1. 页面初次进入先尝试恢复本地缓存。
2. 管理员点击“开始分析”或“重新分析”才发起请求。
3. 返回结果会写入 `localStorage` 缓存。

这样设计的原因是：

1. 避免频繁切页时重复调用大模型。
2. 降低接口耗时对页面体验的影响。
3. 没有配置模型时仍然可以保留上一次分析结果。

### 4.13 方案优点与边界

#### 优点

1. 对外表现为 AI 分析，但底层具备可靠的规则兜底。
2. 输出结构稳定，前端渲染安全。
3. 可兼容 OpenAI、通义千问等 OpenAI 风格接口。
4. Prompt 可控，模型输出可控，结果可解释。
5. 与后台经营看板共用数据口径，减少数字不一致。

#### 当前边界

1. 预测逻辑仍是基于增长率的规则外推，不是时间序列训练模型。
2. 当前 AI 分析主要使用订单、库存、热销等经营数据，尚未把更细粒度的用户行为统计作为显式特征输入。
3. 默认配置下模型是关闭的，需要显式配置 `config/ai.yml` 或环境变量后才能使用真实大模型。

## 6. 答辩可讲亮点

1. 推荐模块不是简单热门排序，而是多路召回与融合排序。
2. 推荐算法同时使用了协同过滤、内容相似、规则重排和热门兜底。
3. 不同推荐策略有明确的权重设计和归一化公式。
4. 推荐结果不仅有分数，还有策略与理由，具备可解释性。
5. AI 数据分析不是“直接把数据丢给模型”，而是先由看板服务聚合，再通过固定 Prompt 和 JSON 校验输出稳定结构。
6. 大模型调用使用 OpenAI 兼容协议，因此可切换 OpenAI、通义千问或其他兼容厂商。
7. 即使模型不可用，系统仍可通过规则分析稳定输出结果，工程可靠性更高。
