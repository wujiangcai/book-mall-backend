#### 1. 订单发货状态验证不一致

**冲突描述：**
- **需求文档**：管理员标记订单发货时，验证订单状态为"已支付(1)"
- **AI编码规范**：只有"待发货(2)"状态的订单可以发货
- **设计文档**：订单发货流程验证订单状态为"已支付(1)"

**影响范围：**
- 后台订单管理模块
- 订单状态流转逻辑
- 订单发货接口实现

**根本原因：**
订单支付后的状态变更逻辑不明确：
- 方案A：支付后保持"已支付(1)"，管理员直接从"已支付"发货
- 方案B：支付后自动变为"待发货(2)"，管理员从"待发货"发货

**推荐方案：方案B（更符合业务逻辑）**

支付成功后应该自动将订单状态从"已支付(1)"变更为"待发货(2)"，理由：
1. 状态语义更清晰："已支付"表示支付完成，"待发货"表示等待发货
2. 管理员操作更明确：只需处理"待发货"状态的订单
3. 状态流转更完整：待支付 → 已支付 → 待发货 → 已发货 → 已完成

**修复步骤：**

1. **修改需求文档 (docs/requirements.md)**
   - 位置：需求 11 - 后台订单管理，验收标准第5条
   - 修改前：`WHEN 管理员标记订单发货，THE System SHALL 验证订单状态为已支付（1）`
   - 修改后：`WHEN 管理员标记订单发货，THE System SHALL 验证订单状态为待发货（2）`
   - 添加说明：`订单支付成功后，系统应自动将订单状态从"已支付(1)"变更为"待发货(2)"`

2. **修改需求文档 (docs/requirements.md)**
   - 位置：需求 8 - 订单创建与支付，验收标准
   - 添加新的验收标准：
   ```
   15. WHEN 用户支付订单成功，THE System SHALL 自动更新订单状态为待发货（2）
   16. THE System SHALL 在支付成功后立即执行状态变更，无需人工干预
   ```

3. **修改设计文档 (docs/design.md)**
   - 位置：6.5 订单支付流程
   - 修改步骤：
   ```
   3. 后端处理：
      - 查询订单信息
      - 验证订单归属权限（order.userId == 当前用户ID）
      - 验证订单状态为待支付（status == 0）
      - 如果状态不对，抛出 InvalidOrderStatusException
      - 更新订单状态为已支付（status = 1）
      - 记录支付时间（pay_time = NOW()）
      - 立即更新订单状态为待发货（status = 2）  // 新增
   ```

4. **修改设计文档 (docs/design.md)**
   - 位置：6.6 订单发货流程
   - 修改前：`验证订单状态为已支付（status == 1）`
   - 修改后：`验证订单状态为待发货（status == 2）`
   - 修改前：`更新订单状态为已发货（status = 3）`
   - 修改后：保持不变

#### 2. 密码加密哈希值不正确

**冲突描述：**
数据库脚本中的 BCrypt 哈希值格式不正确，存在重复字符串。

**当前值：**
```sql
'$2a$10$N.zmdr9k7uOCQb376NoUnuTJ8iAt6Z5EHsM8lE9lBOsl7iAt6Z5EH'
```

**问题：**
- 哈希值中 `iAt6Z5EH` 重复出现两次
- 这不是有效的 BCrypt 哈希格式
- 会导致密码验证失败

**修复方案：**

使用真实的 BCrypt 加密值（需要实际生成）：

```sql
-- 管理员账户（密码：admin123）
-- 使用在线工具或代码生成真实的 BCrypt 哈希
-- 示例（需要替换为实际生成的值）：
'$2a$10$N.zmdr9k7uOCQb376NoUnuTJ8iAt6Z5EHsM8lE9lBOsl7iKLU.yvC'

-- 测试用户（密码：user123）
-- 示例（需要替换为实际生成的值）：
'$2a$10$5K8p7VX3qhJjR8L9mN2pOeX8F9G1H2J3K4L5M6N7O8P9Q0R1S2T3U'
```

**生成方法：**

方法1：使用 Java 代码生成
```java
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

public class PasswordGenerator {
    public static void main(String[] args) {
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
        System.out.println("admin123: " + encoder.encode("admin123"));
        System.out.println("user123: " + encoder.encode("user123"));
    }
}
```

方法2：使用在线工具
- 访问：https://bcrypt-generator.com/
- 输入密码，选择 rounds=10
- 复制生成的哈希值

**修复步骤：**

1. **生成正确的哈希值**
2. **修改数据库脚本 (src/main/resources/sql/schema.sql)**
3. **修改数据库指南 (docs/database-guide.md)** 中的示例哈希值
4. **添加注释说明** 这些是示例值，生产环境必须重新生成

#### 3. ISBN 唯一性约束不明确

**冲突描述：**
- **数据库脚本**：为 ISBN 字段创建了唯一索引 `UNIQUE KEY idx_isbn (isbn)`
- **需求文档**：未提及 ISBN 必须唯一
- **AI编码规范**：未提及 ISBN 唯一性要求

**潜在问题：**
1. 同一本书的不同版本（平装、精装）可能有不同 ISBN
2. 如果 ISBN 为 NULL，多条记录不会违反唯一约束（MySQL 特性）
3. 业务需求不明确

**推荐方案：移除唯一约束**

理由：
1. 图书商城可能需要销售同一本书的不同版本
2. ISBN 可能为空（某些图书没有 ISBN）
3. 如果确实需要防止重复，应该在应用层校验并提示

**修复步骤：**

1. **修改数据库脚本 (src/main/resources/sql/schema.sql)**
   ```sql
   -- 修改前：
   UNIQUE KEY `idx_isbn` (`isbn`),
   
   -- 修改后：
   KEY `idx_isbn` (`isbn`),  -- 改为普通索引，保留查询性能
   ```

2. **修改数据库指南 (docs/database-guide.md)**
   - 在 book 表的索引说明中更新
   - 添加说明：ISBN 使用普通索引而非唯一索引，允许同一 ISBN 的不同版本

3. **修改 AI 编码规范 (.kiro/steering/ai-coding-standards.md)**
   - 在图书管理业务规则中添加：
   ```
   - ISBN 不强制唯一，允许同一本书的不同版本使用不同 ISBN
   - 添加图书时，如果 ISBN 已存在，应提示管理员确认是否为不同版本
   ```

#### 4. 订单状态"已支付"的使用场景模糊

**冲突描述：**
根据推荐的方案B，"已支付(1)"状态应该是一个瞬时状态：
- 支付成功后立即从"待支付(0)"变为"已支付(1)"
- 然后立即变为"待发货(2)"

**问题：**
这样的设计使得"已支付(1)"状态几乎不会被观察到，可能导致：
1. 状态流转日志不完整
2. 前端状态显示闪烁
3. 状态码浪费

**推荐方案：简化状态流转**

**方案A：保留"已支付"作为过渡状态**
```
待支付(0) → 已支付(1) → [自动] → 待发货(2) → 已发货(3) → 已完成(4)
```
- 优点：状态流转完整，便于审计
- 缺点：增加一次数据库更新

**方案B：直接跳过"已支付"**
```
待支付(0) → 待发货(2) → 已发货(3) → 已完成(4)
```
- 优点：简化流程，减少数据库操作
- 缺点：状态语义不够清晰

**推荐：方案A**

保留"已支付"状态，但在支付接口中使用事务一次性完成两次状态更新：

```java
@Transactional
public void payOrder(Long orderId) {
    // 1. 更新为已支付
    orderMapper.updateStatus(orderId, OrderStatus.PAID);
    orderMapper.updatePayTime(orderId, LocalDateTime.now());
    
    // 2. 立即更新为待发货
    orderMapper.updateStatus(orderId, OrderStatus.PENDING_SHIP);
    
    // 3. 记录状态变更日志（可选）
    orderStatusLogMapper.insert(new OrderStatusLog(orderId, 0, 1, "用户支付"));
    orderStatusLogMapper.insert(new OrderStatusLog(orderId, 1, 2, "自动转待发货"));
}
```

**修复步骤：**

1. **在设计文档中明确说明**
   - 位置：6.5 订单支付流程
   - 添加说明：支付成功后在同一事务中完成两次状态更新

2. **在 AI 编码规范中添加实现细节**
   ```
   - 订单支付成功后，在同一事务中完成以下操作：
     1. 更新订单状态为"已支付(1)"
     2. 记录支付时间
     3. 立即更新订单状态为"待发货(2)"
     4. 记录状态变更日志（可选）
   ```

#### 7. 外键约束策略的业务影响

**冲突描述：**
- **数据库脚本**：book 表使用 `ON DELETE RESTRICT`，图书被订单引用时不能删除
- **需求文档**：管理员删除图书时，验证该图书未在待处理订单中
- **AI编码规范**：未明确图书删除策略

**潜在问题：**
如果图书已经在历史订单中，将永远无法删除（即使订单已完成）

**推荐方案：使用软删除**

不物理删除图书，而是通过 status 字段标记为下架：
```
status = 0: 下架（不显示，不可购买，但保留数据）
status = 1: 上架（正常显示和销售）
status = -1: 已删除（软删除，仅管理员可见）
```

**修复步骤：**

1. **在 AI 编码规范中明确删除策略**
   ```markdown
   #### 图书删除策略
   - 图书不应该被物理删除，只能下架或软删除
   - 下架（status=0）：图书不在前台显示，但管理员可以重新上架
   - 软删除（status=-1）：图书标记为已删除，不可恢复，但保留历史数据
   - 物理删除：仅在确认该图书从未被订单引用时才允许
   ```

2. **修改需求文档**
   - 位置：需求 5 - 图书信息管理
   - 修改验收标准第9条：
   ```
   9. WHEN 管理员删除图书，THE System SHALL 使用软删除方式标记图书状态
   10. IF 图书已被订单引用，THEN THE System SHALL 禁止物理删除
   11. THE System SHALL 提供软删除和物理删除两种方式，默认使用软删除
   ```

#### 8. 订单取消后的库存恢复时机

**冲突描述：**
- **需求文档**：用户取消订单时恢复图书库存
- **设计文档**：未明确库存扣减和恢复的时机
- **AI编码规范**：创建订单时扣减库存

**问题：**
如果创建订单时就扣减库存，但用户未支付就取消，会导致：
1. 库存被占用但没有实际销售
2. 恶意用户可能通过创建大量订单占用库存

**推荐方案：支付后扣减库存**

```
方案A：创建订单时扣减（当前方案）
- 优点：防止超卖
- 缺点：库存可能被占用

方案B：支付后扣减（推荐）
- 优点：库存利用率高
- 缺点：可能出现超卖（需要在支付时再次校验）

方案C：混合方案
- 创建订单时预占库存（标记为"锁定"）
- 支付后正式扣减
- 超时未支付自动释放锁定库存
```

**推荐：方案B + 支付时二次校验**

**修复步骤：**

1. **在 AI 编码规范中明确库存扣减策略**
   ```markdown
   #### 库存扣减策略（推荐方案）
   
   **创建订单时：**
   - 仅校验库存是否充足，不扣减库存
   - 记录订单中的商品和数量
   
   **支付订单时：**
   - 再次校验库存是否充足（防止并发超卖）
   - 使用乐观锁扣减库存：`UPDATE book SET stock = stock - ? WHERE id = ? AND stock >= ?`
   - 如果库存不足，支付失败，订单状态保持"待支付"
   
   **取消订单时：**
   - 如果订单已支付，恢复库存
   - 如果订单未支付，无需恢复（因为未扣减）
   ```

2. **修改需求文档**
   - 位置：需求 8 - 订单创建与支付
   - 修改验收标准：
   ```
   8. WHEN 用户支付订单，THE System SHALL 扣减图书库存
   9. WHEN 用户支付订单，THE System SHALL 再次校验库存充足性
   10. IF 支付时库存不足，THEN THE System SHALL 返回库存不足错误，订单保持待支付状态
   ```

       │                     │ 自动变更（同一事务）
       │                     ↓
       │              ┌─────────────┐
       │              │  待发货(2)   │ ← 等待管理员发货

       │                     │
       │                     │ 管理员发货
