## 第 2 页：项目背景与意义

### 页面放什么

- 传统图书销售/管理存在的问题
- 为什么要做图书商城系统
- 这个系统的应用价值

### 建议文案

- 传统图书管理方式效率较低，信息更新不及时。
- 用户在线购书需求增加，需要便捷的浏览、下单与支付流程。
- 管理员需要统一管理图书、分类、订单和用户信息。
- 因此设计并实现一个前后端分离的图书商城系统具有现实意义。

### 你怎么讲

> 本项目的背景是传统图书销售和管理方式效率较低，用户和管理员都缺少统一、高效的信息化平台。因此我设计了一个集图书展示、购物车、订单支付与后台管理于一体的图书商城系统，提高业务处理效率和管理规范性。

### 页面放什么

- 用户端目标

## 第 4 页：系统总体架构

### 页面放什么

建议放一张结构图，结构从上到下：

- Vue 前端
- HTTP/REST API
- Spring Boot 后端
- MyBatis
- MySQL

右侧可标注：

- JWT 认证
- 支付宝支付
- 腾讯云 COS

### 你怎么讲

> 本系统采用前后端分离架构。前端基于 Vue 3 构建用户端和管理端页面，后端基于 Spring Boot 提供 RESTful API，数据通过 MyBatis 访问 MySQL 数据库。同时系统集成了 JWT 认证、支付宝沙箱支付和腾讯云 COS 图片上传能力。

### 页面放什么

| 层次 | 技术 | 作用 |
|---|---|---|

## 第 8 页：用户认证流程

### 页面放什么

流程图：

登录 -> 校验账号密码 -> 生成 JWT -> 前端保存 Token -> 请求自动携带 Token -> 拦截器解析 -> 获取 userId/role

### 你怎么讲

> 用户登录成功后，后端会生成 JWT Token 返回给前端。前端后续请求会自动携带该 Token，后端通过拦截器解析 Token 并识别当前用户身份。后台接口还会额外校验角色是否为管理员。

## 第 9 页：核心业务流程——下单流程

### 页面放什么

流程图建议：

浏览图书 -> 加入购物车 -> 选择地址 -> 创建订单 -> 生成订单主表/明细 -> 删除购物车项 -> 返回订单信息

### 你怎么讲

> 用户从购物车提交下单请求后，系统会校验地址归属、购物车项合法性、图书状态和库存情况，然后生成订单主表和订单明细表，并删除已提交的购物车项，从而完成订单创建。

---

## 第 10 页：核心业务流程——支付流程

### 页面放什么

建议流程图：

用户点击支付 -> 后端生成支付宝表单 -> 跳转支付宝 -> 支付完成 -> 异步通知/同步回跳 -> 验签 -> 更新订单状态 -> 扣库存

### 你怎么讲

> 支付流程中，后端会调用支付宝 SDK 生成支付表单。支付完成后，系统主要依靠支付宝异步通知确认支付成功，然后完成订单状态更新和库存扣减。同步回跳主要用于前端页面展示。

---

## 第 11 页：核心实现——订单与库存一致性

### 页面放什么

标题下列 3 个关键词：

- 乐观锁 version
- 幂等处理
- 状态流转控制

### 建议文案

- 图书表和订单表加入 `version` 字段
- 更新时带版本号，成功后版本自增
- 防止并发更新覆盖
- 防止支付回调重复处理
- 防止库存超卖

### 你怎么讲

> 为了保证订单与库存的一致性，我在订单表和图书表中加入了 version 字段，并基于乐观锁实现更新控制。同时，支付回调处理增加了幂等判断，避免重复通知导致重复扣库存或重复更新状态。

---

## 第 12 页：核心实现——统一异常处理

### 页面放什么

- 统一响应结构 `Result`
- 全局异常处理器 `GlobalExceptionHandler`
- 常见错误码：400、401、403、404、500

### 你怎么讲

> 为了提高接口规范性，系统使用统一响应对象封装返回结果，并通过全局异常处理器统一处理业务异常、权限异常和参数异常，减少重复代码并提高前后端联调效率。

## 最后给你的实用建议

### 1. PPT 不要做太花

- 白底或浅色底就行
- 一页只讲一个重点
- 少放大段文字，多放流程图和结构图

### 2. 演示顺序固定

- 先讲背景和架构
- 再讲数据库和功能
- 再讲订单支付与亮点
- 最后讲不足与总结

### 3. 你最该背熟的页

- 第 4 页：总体架构
- 第 7 页：数据库设计
- 第 9 页：下单流程
- 第 10 页：支付流程
- 第 11 页：乐观锁
- 第 16 页：项目亮点

### 4. 如果时间不够

至少讲清楚这 4 件事：

- 项目做了什么
- 用了什么技术
- 订单支付怎么走
- 乐观锁为什么要做
