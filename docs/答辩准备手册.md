## 1. 先用一句话讲清项目

这个项目是一个**前后端分离的图书商城管理系统**。

- 前台面向普通用户：注册登录、浏览图书、加入购物车、管理地址、下单、支付、查看订单。
- 后台面向管理员：登录后台、管理图书、分类、用户、订单、轮播图，并查看数据看板。

如果老师让你**30 秒介绍项目**，你可以直接这样说：

> 我的毕业设计是一个前后端分离的图书商城系统。前端使用 Vue 3 构建用户端和管理端界面，后端使用 Spring Boot 提供 RESTful API，数据库使用 MySQL，数据访问层使用 MyBatis。系统实现了用户登录注册、图书展示、购物车、地址管理、订单创建、支付宝支付、后台订单与图书管理、首页轮播图以及后台数据统计等功能。

## 3. 技术栈总览

这部分是答辩高频问题。

### 3.1 后端技术栈

- `Spring Boot`：后端主框架，用来快速搭建 Web 应用。
- `Spring MVC`：处理请求和响应。
- `Spring Security`：提供安全能力，这个项目主要用到了密码加密和基础安全配置。
- `MyBatis`：操作数据库，SQL 写在 XML 中，更直观。
- `MySQL`：存储业务数据。
- `JWT`：用户登录成功后生成 Token，用来做无状态认证。
- `Lombok`：简化 Java 实体类代码。
- `SpringDoc / Swagger`：自动生成接口文档。
- `支付宝沙箱 SDK`：实现支付流程。
- `腾讯云 COS SDK`：实现轮播图和图书封面的对象存储上传。

### 3.2 前端技术栈

- `Vue 3`：前端框架。
- `TypeScript`：让前端代码更规范，类型更清晰。
- `Vite`：前端构建工具，启动快。
- `Vue Router`：管理页面跳转。
- `Pinia`：管理登录状态等全局数据。
- `Axios`：调用后端接口。
- `Arco Design Vue`：前端 UI 组件库。

### 3.3 你可以这样回答“为什么选这些技术”

- 选择 `Spring Boot` 是因为它生态成熟、配置方便，适合中小型管理系统开发。
- 选择 `MyBatis` 是因为商城系统里查询条件较多，SQL 写在 XML 中更容易控制。
- 选择 `Vue 3 + Vite` 是因为开发效率高、页面组件化清晰。
- 选择 `JWT` 是因为前后端分离项目通常采用无状态认证，便于接口调用。

## 4. 系统总体架构

你可以把整个项目理解成 5 层。

### 4.1 架构分层

1. **表现层（前端页面）**
   - 用户看到的页面。
   - 包括前台页面和后台页面。

2. **控制层（Controller）**
   - 接收前端请求。
   - 做参数接收、基础校验。
   - 调用业务层。

3. **业务层（Service）**
   - 处理业务逻辑。
   - 例如：创建订单、扣库存、修改状态。

4. **数据访问层（Mapper）**
   - 通过 MyBatis 执行 SQL。
   - 负责从数据库查询和保存数据。

5. **数据库层（MySQL）**
   - 存储用户、图书、订单等数据。

### 4.2 一次请求是怎么走的

以“用户查看订单列表”为例：

1. 前端页面发请求。
2. Axios 自动带上 Token。
3. 后端拦截器解析 Token，拿到 `userId`。
4. Controller 接收请求。
5. Service 处理分页和业务逻辑。
6. Mapper 执行 SQL 查询订单。
7. 数据库返回结果。
8. 后端统一包装成 `Result` 返回前端。
9. 前端页面渲染出来。

### 4.3 一句话解释“前后端分离”

> 前端只负责页面展示和交互，后端只负责业务逻辑和数据处理，二者通过 HTTP 接口通信。

## 6. 核心功能模块说明

把系统拆成 8 个模块就很好讲。

### 6.1 用户认证模块

包含：

- 用户注册
- 用户登录
- 管理员登录
- 获取个人信息
- 修改个人信息
- 修改密码

核心实现思路：

- 用户注册时，后端先检查用户名和手机号是否重复。
- 密码不会明文存库，而是使用 `BCrypt` 加密后保存。
- 登录成功后，后端生成 JWT Token 返回前端。
- 前端把 Token 保存到浏览器本地存储中。
- 以后每次请求都自动把 Token 放到 `Authorization` 请求头中。

### 6.2 图书模块

前台功能：

- 图书分页查询
- 按分类筛选
- 按关键词搜索
- 按价格区间筛选
- 查看图书详情

后台功能：

- 新增图书
- 修改图书
- 删除图书（软删除）
- 上下架管理

关键点：

- 图书删除不是直接从数据库里删，而是把 `status` 设为 `-1`，这种做法叫**软删除**。
- 这样做的好处是数据可追踪，避免误删后无法恢复。

### 6.3 分类模块

- 前台读取启用状态的分类树。
- 后台可以新增、修改、禁用、删除分类。
- 图书和分类是关联的，因此图书新增/修改时要检查分类是否合法。

### 6.4 购物车模块

- 添加商品到购物车
- 修改购物车商品数量
- 删除购物车项
- 查看购物车列表

关键点：

- 加入购物车前会检查图书是否存在、是否上架、库存是否足够。
- 如果购物车已存在该图书，不会重复插入，而是累加数量。

### 6.5 地址模块

- 新增地址
- 修改地址
- 删除地址
- 设置默认地址
- 查看地址列表

业务意义：

- 地址信息独立成表，便于一个用户拥有多个地址。
- 下单时，订单会关联地址，方便后续配送和订单展示。

### 6.6 订单模块

前台：

- 创建订单
- 支付订单
- 取消订单
- 查看订单列表
- 查看订单详情

后台：

- 查看订单列表
- 查看订单详情
- 修改订单状态
- 取消订单
- 发货
- 代客创建订单

这个模块是整个项目的重中之重。

### 6.7 支付模块

- 使用支付宝沙箱实现支付联调。
- 前端调用支付接口后，后端返回一段 HTML 表单。
- 用户跳到支付宝支付页面完成支付。
- 支付完成后，支付宝会发两类通知：
  - 同步跳转：让用户浏览器跳回系统页面
  - 异步通知：服务器到服务器通知，真正可靠

项目中真正以**异步通知**作为支付成功依据。

### 6.8 后台数据看板模块

包含：

- 总览数据
- 最近 7 天订单趋势
- 订单状态分布
- 分类统计
- 热门图书
- 低库存图书
- 最近订单

作用：

- 让系统不仅能“增删改查”，还能提供简单运营视角。

## 8. 订单状态流转，一定要会背

订单状态是老师非常喜欢问的点。

本项目中订单状态定义如下：

- `0`：待支付（UNPAID）
- `1`：已支付（PAID）
- `2`：待发货（PENDING_SHIP）
- `3`：已发货（SHIPPED）
- `4`：已完成（COMPLETED）
- `5`：已取消（CANCELLED）

虽然常量里保留了 `PAID`，但从当前业务实现看，支付成功后主要直接流转到**待发货**，因为这更适合商城实际业务。

### 8.1 正常流转

`待支付 -> 待发货 -> 已发货 -> 已完成`

### 8.2 异常流转

`待支付 -> 已取消`

### 8.3 为什么不能乱跳状态

因为订单状态代表真实业务阶段。

例如：

- 待支付不能直接变成已发货
- 已完成不能再取消
- 已取消不能再支付

所以项目在后台修改订单时加了**状态机规则校验**。

你可以这样回答：

> 为了保证订单数据一致性，我在订单服务中限制了状态跳转规则，避免后台人员误操作造成业务状态错乱。

## 9. 登录认证流程，老师很爱问

### 9.1 登录流程

1. 用户输入用户名和密码。
2. 前端调用登录接口。
3. 后端查询数据库中的用户。
4. 使用 `BCrypt` 比较输入密码和数据库加密密码。
5. 如果匹配成功，则生成 JWT Token。
6. 前端保存 Token。
7. 之后访问受保护接口时自动携带 Token。

### 9.2 JWT 里放了什么

主要放了两个字段：

- `userId`
- `role`

目的：

- `userId` 用来区分当前是谁。
- `role` 用来区分普通用户和管理员。

### 9.3 后端如何校验 Token

项目里没有完全依赖 Spring Security 默认登录流程，而是用了**自定义拦截器**：

- `UserAuthInterceptor`
- `AdminAuthInterceptor`

工作方式：

1. 从请求头读取 `Authorization`
2. 用 `JwtUtil` 解析 Token
3. 解析出 `userId` 和 `role`
4. 保存到 `request attribute`
5. Controller 再从 request 中取出当前用户信息

### 9.4 为什么后台还要单独校验 role

因为普通用户登录后也有 Token，但不能访问后台接口。

所以管理员拦截器会额外判断：

- `role == ADMIN` 才允许进入后台 API

## 10. 下单流程，重点中的重点

下面是最推荐你在答辩里讲清楚的一条业务链。

### 10.1 用户下单流程

1. 用户先把图书加入购物车。
2. 用户维护好收货地址。
3. 用户提交 `cartIds + addressId` 创建订单。
4. 后端校验：
   - 地址是否属于当前用户
   - 购物车项是否存在
   - 图书是否存在
   - 图书是否上架
   - 库存是否足够
5. 后端生成订单主表 `order`
6. 后端生成订单明细表 `order_item`
7. 后端删除已下单的购物车项
8. 返回订单创建结果给前端

### 10.2 为什么创建订单时不立刻扣库存

这是个很值得讲的设计点。

项目当前实现选择：

- **创建订单时只校验库存**
- **支付成功回调时才真正扣库存**

原因：

- 如果一创建订单就扣库存，但用户一直不支付，会导致库存被长期占用。
- 在支付成功后再扣库存，更符合实际交易闭环。

当然，这种设计也有代价：

- 如果支付瞬间库存被别人抢完，需要在支付回调阶段再次校验并扣减库存。

而项目确实这样做了。

## 11. 支付流程，必须搞明白

### 11.1 支付发起流程

1. 用户点击“支付订单”。
2. 前端调用 `/api/front/orders/{id}/pay`。
3. 后端检查：
   - 订单是否存在
   - 订单是否属于当前用户
   - 是否还是待支付状态
   - 图书是否仍存在、上架且库存足够
4. 后端调用支付宝 SDK 生成支付表单 HTML。
5. 前端接收到 HTML 后跳转到支付宝页面。

### 11.2 支付成功后发生什么

支付宝会通知系统两次：

#### 方式一：异步通知

- 支付宝服务器主动调用系统后端通知地址。
- 这是最可靠的支付成功依据。

#### 方式二：同步回跳

- 用户浏览器跳回商城页面。
- 这只是给用户看的，不能作为唯一支付成功依据。

### 11.3 为什么异步通知更可靠

因为同步回跳依赖用户浏览器：

- 用户可能中途关闭页面
- 网络可能中断
- 页面跳回可能失败

但异步通知是服务器之间通信，更稳定。

所以你可以这样说：

> 系统将支付宝异步通知作为支付成功的最终确认依据，同步回跳仅用于前端页面展示和用户体验补充。

### 11.4 支付回调后系统做了什么

1. 验签，确认通知来自支付宝。
2. 校验 appId、sellerId、支付金额是否和订单一致。
3. 校验订单当前状态仍然是待支付。
4. 更新订单状态为待发货，并写入支付时间和交易号。
5. 查询订单明细。
6. 对每一项图书执行库存扣减。

## 12. 为什么要做乐观锁

这是本项目一个明显的技术亮点。

### 12.1 什么是乐观锁

可以把它理解成：

> 修改数据时，不直接假设没人改过，而是先检查“版本号有没有变”。

如果版本号没变，说明没人抢先修改，可以更新。

如果版本号变了，说明有人先改了，这次更新就失败。

### 12.2 本项目哪里用了乐观锁

- `book` 表有 `version`
- `order` 表有 `version`

### 12.3 为什么这里要用乐观锁

典型场景：

#### 场景一：多人同时购买同一本库存紧张的书

- 两个人几乎同时支付成功
- 如果不做并发控制，可能都扣库存成功，造成超卖

#### 场景二：订单支付回调重复通知

- 第一次通知刚处理完
- 第二次通知又来了
- 如果没有版本控制和幂等逻辑，就可能重复修改订单状态甚至重复扣库存

### 12.4 代码里是怎么实现的

以库存扣减为例，SQL 思路类似这样：

- 更新时带上旧版本号
- 只有 `version` 匹配且 `stock >= quantity` 才更新成功
- 更新成功后 `version = version + 1`

如果更新行数为 0，说明：

- 可能库存不够
- 也可能有人先修改过了

此时系统会重试几次，仍失败则提示“数据已变更，请刷新后重试”。

### 12.5 这有什么答辩价值

这是你可以主动强调的地方：

> 为了提升订单支付和库存变更场景下的数据一致性，我在订单表和图书表中加入了 version 字段，并基于 MyBatis SQL 实现了乐观锁更新机制，用来应对并发修改和重复回调问题。

## 13. 为什么要统一响应和统一异常处理

### 13.1 统一响应 `Result`

所有接口基本都返回类似格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1710000000000

```

好处：

- 前端处理简单
- 成功失败结构一致
- 容易扩展

### 13.2 全局异常处理

如果没有全局异常处理，会出现很多问题：

- 每个 Controller 都要自己 try-catch
- 错误格式不统一
- 前端难以处理

现在项目通过 `GlobalExceptionHandler` 统一处理：

- 业务异常
- 参数校验异常
- 401 未登录
- 403 无权限
- 404 数据不存在
- 500 服务器异常

你可以这样说：

> 我通过全局异常处理器统一封装错误返回，提升了接口规范性，也降低了控制层重复代码。

## 18. 老师追问“为什么这样设计”时的标准回答

### 18.1 为什么用 MyBatis，不用 JPA

> 因为商城系统中筛选、统计和多表查询场景较多，MyBatis 对 SQL 控制更直接，便于根据业务需求精细编写查询。

### 18.2 为什么用 JWT

> 因为前后端分离项目不适合依赖传统 Session，JWT 可以让服务端保持无状态，接口调用更方便。

### 18.3 为什么密码要加密

> 出于安全考虑，数据库不能存明文密码，所以使用 BCrypt 进行哈希加密。

### 18.4 为什么要做软删除

> 软删除可以保留历史数据，避免误删后无法恢复，也便于审计和后续业务兼容。

### 18.5 为什么支付成功后才扣库存

> 这样可以避免用户创建订单后长时间不支付而占用库存，更符合真实业务流程。

### 18.6 为什么要做乐观锁

> 为了防止并发更新导致的数据覆盖、重复扣库存和重复支付处理，提升数据一致性。

## 22. 最容易被问到的 20 个答辩问题

下面我直接给你“问题 + 可背答案”。

### 问题 1：你的项目属于什么类型？

答：

> 这是一个前后端分离的图书商城管理系统，包含用户端和管理员端两部分。

### 问题 2：系统核心功能有哪些？

答：

> 核心功能包括用户注册登录、图书浏览、购物车、地址管理、订单创建、支付宝支付、后台图书管理、订单管理、用户管理和数据看板。

### 问题 3：你为什么选择前后端分离？

答：

> 前后端分离可以让页面展示和业务逻辑解耦，开发和维护更清晰，也更符合当前实际开发模式。

### 问题 4：JWT 的作用是什么？

答：

> JWT 用于无状态认证。登录成功后，后端生成 Token，前端后续请求携带 Token，后端解析后识别当前用户身份。

### 问题 5：为什么不用 Session？

答：

> 因为前后端分离场景下，JWT 更适合接口调用，也减少了服务端对会话状态的依赖。

### 问题 6：购物车模块的意义是什么？

答：

> 购物车起到暂存待购商品的作用，用户可以先选购多本图书，再统一下单，提高体验。

### 问题 7：订单和订单明细为什么分两张表？

答：

> 因为一张订单可能包含多本图书，所以订单主表保存整体信息，订单明细表保存每一本图书的信息，这是典型的一对多设计。

### 问题 8：什么是软删除？

答：

> 软删除就是不真正删除数据库记录，而是通过状态字段标记为已删除，这样便于数据恢复和追溯。

### 问题 9：为什么密码不能明文存储？

答：

> 明文密码泄露风险很高，所以项目使用 BCrypt 加密后再存库。

### 问题 10：为什么统一响应格式？

答：

> 统一响应格式可以让前后端交互更规范，前端处理成功和失败的逻辑也更简单。

### 问题 11：为什么要做全局异常处理？

答：

> 可以减少重复 try-catch，统一错误返回格式，提高代码可维护性。

### 问题 12：支付成功后你是怎么确认的？

答：

> 主要通过支付宝异步通知确认支付结果，同步回跳只用于辅助页面展示。

### 问题 13：为什么支付成功后才扣库存？

答：

> 为了避免用户创建订单但不付款而长期占用库存，因此在支付成功后再正式扣减库存。

### 问题 14：为什么要做乐观锁？

答：

> 为了保证订单状态更新和库存扣减时的数据一致性，防止并发场景下的数据覆盖和超卖。

### 问题 15：项目里有哪些安全措施？

答：

> 主要包括密码加密、JWT 鉴权、前后台分角色拦截、统一异常处理、敏感日志脱敏等。

### 问题 16：后台和前台权限怎么区分？

答：

> 通过 JWT 中的 role 字段区分，并由不同拦截器拦截对应接口。

### 问题 17：为什么选 MyBatis？

答：

> 因为项目查询较多、筛选条件较复杂，MyBatis 对 SQL 控制更灵活。

### 问题 18：项目测试做了哪些？

答：

> 目前主要对订单服务做了单元测试，重点覆盖支付回调、状态校验、库存扣减与并发重试等核心逻辑。

### 问题 19：项目还有哪些可以优化？

答：

> 可以进一步引入 Redis 缓存、消息队列、订单超时机制、细粒度 RBAC 权限和更完善的自动化测试。

### 问题 20：你认为这个项目最有价值的地方是什么？

> 我认为最有价值的是它不仅实现了完整业务闭环，还考虑了支付回调、并发控制、统一异常处理和后台数据统计这些更贴近实际工程场景的问题。
